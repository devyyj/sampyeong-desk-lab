# 삼평동 책상 전략 연구소 (SDL) 서비스 진단 및 수정 계획

본 문서는 삼평동 책상 전략 연구소(SDL) 서비스가 "빈 껍데기"처럼 동작하는 현상에 대한 진단 결과와 이를 정상적으로 동작하도록 만들기 위한 수정 계획을 서술합니다. **본 프로젝트의 개발 지침에 따라 모든 개발은 TDD(테스트 주도 개발)를 원칙으로 진행됩니다.**

---

## 1. 전체 기능 진단 결과

현재 서비스는 UI 레이아웃과 일부 인증 흐름(가입/로그인)의 외형은 갖추어져 있으나, 핵심 비즈니스 로직과 데이터 흐름이 데이터베이스(DB)와 연동되지 않고 클라이언트 상태(State) 또는 더미 API 응답에 의존하고 있어 새로고침 시 데이터가 모두 유실되거나 화면에 아무것도 표시되지 않는 상태입니다.

### 🔴 핵심 문제점 상세

| 영역 | 현상 | 원인 | 영향 |
| :--- | :--- | :--- | :--- |
| **메인 피드 (Feed)** | 게시글 목록이 비어 있음. 게시글 작성, 좋아요, 댓글 작성 시 새로고침하면 사라짐. | 1. `/api/mock-data` API의 `feed` 액션이 빈 배열(`[]`)을 반환함.<br>2. 피드 페이지(`src/app/page.tsx`)의 작성/좋아요/댓글 로직이 API 호출 없이 React State만 조작함. | 사용자가 게시글을 작성하거나 소통할 수 없음. |
| **공지사항 (Announcements)** | 공지글 목록이 비어 있음. 공지 작성/삭제가 새로고침 시 유지되지 않음. | 1. `/api/mock-data` API의 `announcements` 액션이 빈 배열(`[]`)을 반환함.<br>2. 관리자 작성/삭제 로직이 로컬 State만 업데이트함. | 중요 공지사항 전파 불가. |
| **프로필 편집 (Profile Edit)** | 프로필 수정 페이지에서 정보를 변경하고 저장해도 반영되지 않음. | 1. `profile/edit/page.tsx`의 저장(`handleSubmit`) 로직이 실제 API 호출이나 DB 업데이트 없이 성공 메시지만 띄우고 리다이렉트함. | 사용자 프로필 정보(이름, 부서, 자기소개 등) 변경 불가. |
| **회원 탈퇴 (Withdraw)** | 회원 탈퇴를 진행해도 실제 계정이 삭제되지 않음. | 1. 탈퇴 버튼 클릭 시 세션 쿠키만 지울 뿐, Supabase Auth 및 `profiles` 테이블에서 실제 계정을 삭제하지 않음. | 탈퇴 후 동일 닉네임 재가입 불가 등 오동작 유발. |
| **데이터베이스 연동** | Drizzle 스키마가 존재하나 실제 데이터 쓰기/읽기(일부 예외)에 전혀 사용되지 않음. | 1. `/api/mock-data`에서 `profiles` 조회 시에만 DB를 조회하고, 그 외 게시글, 댓글, 좋아요 테이블은 읽기/쓰기 코드가 구현되지 않음. | 시스템 전반의 영속성(Persistence) 결여. |

## 2. TDD 기반 수정 계획

모든 수정은 **실패하는 테스트를 먼저 작성(Red)**하고, 이를 **통과하는 최소한의 코드를 작성(Green)**한 뒤, **리팩토링(Refactor)**하는 TDD 사이클을 준수하여 진행합니다.

### 1단계: API 라우트 연동 및 DB CRUD 구현 (TDD)
- **대상**: `/api/mock-data/route.ts` (또는 필요시 전용 API 라우트 분리)
- **테스트 케이스 구현 (`tests/api-routes.test.ts` 등)**:
  - `feed` 요청 시 DB의 `posts`, `profiles`, `likes`, `comments`를 정상적으로 조인하여 반환하는지 검증하는 테스트 작성
  - `add-post` 요청 시 실제 DB에 `posts` 레코드가 생성되는지 검증하는 테스트 작성
  - `like-toggle`, `comment-add`, `comment-delete` 등 각각의 API 액션 동작을 검증하는 테스트 작성
- **기능 구현**: Drizzle ORM을 사용하여 DB 스키마(`posts`, `comments`, `likes`, `tags` 등)에 맞는 CRUD 쿼리 작성 및 API 연동

### 2단계: 메인 피드 페이지 (`src/app/page.tsx`) 연동
- **대상**: 피드 조회, 게시글 작성, 좋아요 토글, 댓글 작성 및 삭제 기능
- **테스트 케이스 구현 (`src/app/page.test.tsx` 등)**:
  - 페이지 로드 시 API를 호출하여 게시글 목록을 가져오는지 검증
  - 작성 버튼 클릭 시 등록 API가 호출되고 목록이 갱신되는지 검증
  - 좋아요 및 댓글 등록/삭제 API 호출 후 UI 상태가 동기화되는지 검증
- **기능 구현**: 로컬 State 조작 코드를 `fetch` API 호출 및 상태 동기화 로직으로 변경

### 3단계: 공지사항 페이지 (`src/app/announcements/page.tsx`) 연동
- **대상**: 공지사항 조회, 관리자 권한 검증, 공지 등록 및 삭제
- **테스트 케이스 구현**:
  - `admin` 권한을 가진 사용자 세션에서만 공지 작성 폼이 노출되는지 검증
  - 공지 등록 시 DB에 저장되고 최상단에 노출되는지 검증
- **기능 구현**: Supabase Session 정보에 따른 관리자(`role: 'admin'`) 분기 처리 공고히 하고 API와 연동

### 4단계: 프로필 편집 및 회원 탈퇴 구현
- **대상**: 프로필 수정 (`src/app/profile/edit/page.tsx`), 회원 탈퇴
- **테스트 케이스 구현**:
  - 프로필 수정 폼 제출 시 API를 통해 `profiles` 테이블 및 Supabase Auth `user_metadata`가 갱신되는지 검증
  - 회원 탈퇴 요청 시 DB 데이터 지우고 Supabase Auth 사용자 삭제 API가 호출되는지 검증
- **기능 구현**: `/api/mock-data` (또는 `/api/profile`) 수정 처리 작성 및 연동
