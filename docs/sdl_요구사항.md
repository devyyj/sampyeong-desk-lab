# 삼평동 책상 전략 연구소 (sdl) 웹 서비스 요구사항 정의서

## 1. 개요
* **서비스명**: 삼평동 책상 전략 연구소 웹 서비스 (삼책연 / sdl)
* **목적**: 사내 보드게임 동아리 회원들의 활발한 소통, 회원 명부 관리, 공지사항 및 피드형 자유 게시판을 통한 정보 공유 및 커뮤니티 활성화.
* **주요 UI/UX 컨셉**: 
  * 인스타그램 스타일의 모던하고 직관적인 피드 및 모바일 우선(Mobile-First) UI.
  * 가입 장벽을 최소화하기 위한 최소한의 개인정보 수집 및 단순화된 로그인 절차.

---

## 2. 시스템 아키텍처 및 기술 스택 (제안)
* **Frontend**: Next.js (App Router)
* **Backend & Database**: Supabase (PostgreSQL)
* **Authentication**: Supabase Auth (커스텀 닉네임 + 4자리 숫자 패스워드 로그인 처리)
* **Storage**: Supabase Storage (프로필 이미지 및 게시글 이미지 업로드용)
* **Styling**: Modern Vanilla CSS or TailwindCSS
* **Test Framework**: Jest + React Testing Library (TDD 원칙 준수)

---

## 3. 기능 요구사항 상세

### 3.1. 회원가입 (Sign Up)
* **개인정보 최소화**: 가입 시 필수 입력 항목을 최소화하여 가입 부담을 경감한다.
* **입력 항목**:
  * **닉네임 (Unique)**: 시스템 내 고유 식별 명칭 (중복 검사 필수).
  * **이름 (실명)**: 동아리 내 원활한 소통 및 신원 확인을 위해 필수 입력.
  * **부서 (팀)**: 사내 동아리 특성을 반영하여 필수 입력.
  * **비밀번호**: **숫자 4자리**로 제한.
* **UI/UX 특징**:
  * 비밀번호 입력 필드 포커스 시 화면에 커스텀 **숫자 키패드(가상 키패드)**를 제공하여 마우스 클릭 또는 터치로 입력할 수 있도록 유도한다. (입력값 탈취 방지 및 모바일 편의성 극대화)
* **내부 프로세스**:
  * 비밀번호는 Supabase Auth에서 요구하는 이메일/비밀번호 방식을 우회하거나, `닉네임@sdl.com` 형태의 가상 이메일을 생성하고 비밀번호를 해싱하여 가입 처리하는 방안 검토.

### 3.2. 로그인 (Sign In) / 로그아웃 (Sign Out)
* **입력 항목**: 닉네임, 비밀번호 (숫자 4자리).
* **UI/UX 특징**:
  * 로그인 시에도 비밀번호 입력은 커스텀 **숫자 키패드**를 통해 입력받는다.
* **세션 유지**: 자동 로그인 및 로그인 상태 유지 기능 제공.

### 3.3. 회원 정보 관리 (Profile)
* **수정 가능 항목**: 닉네임(중복 검사), 이름, 부서, 비밀번호(숫자 4자리).
* **추가 프로필 설정**:
  * **자기소개**: 자유 형식의 단문 텍스트 작성 가능.
  * **프로필 사진**: 이미지 업로드 및 자르기(Crop) 기능 제공 (Supabase Storage 저장).
* **회원 탈퇴**: 
  * 탈퇴 시 본인이 작성한 게시글 및 댓글의 보존 여부 결정(선택: 탈퇴자 회원 정보는 삭제 또는 익명화 처리, 작성 게시물은 유지하거나 일괄 삭제 옵션 제공).

### 3.4. 회원 명부 (Directory)
* **기능**: 가입된 전체 동아리 회원의 목록을 확인하는 페이지.
* **표시 정보**: 프로필 사진, 닉네임, 실명, 부서, 자기소개.
* **정렬 및 검색**: 가입일순, 이름순 정렬 및 부서/이름 검색 기능.

### 3.5. 자유 게시판 (Feed)
* **UI/UX**: 인스타그램 스타일의 카드/피드형 레이아웃.
* **스크롤 방식**: **무한 스크롤(Infinite Scroll)**을 적용하여 스크롤 시 데이터를 추가 로드한다.
* **기능**:
  * **게시글 작성/수정/삭제**: 텍스트 및 이미지(선택 사항) 업로드 가능. 본인 글만 수정/삭제 가능.
  * **댓글(Comment)**: 게시글별 댓글 작성, 수정, 삭제 기능.
  * **좋아요(Like)**: 토글 방식의 좋아요 기능. 좋아요 개수 표시 및 좋아요 누른 회원 목록(선택) 제공.
  * **태그(Tag)**: 본문 내 `#태그` 형식의 해시태그를 추출하여 태그별 모아보기 제공 (검색 기능 대체).

### 3.6. 공지 사항 (Announcements)
* **권한 정책**:
  * **작성/수정/삭제**: **관리자(Admin) 권한**을 가진 사용자만 가능.
  * **조회/댓글 작성/수정/삭제**: 모든 일반 회원이 가능.
* **UI/UX**: 게시판 상단 고정 또는 별도 메뉴로 분리하여 가독성 확보.

## 4. 비기능 및 보안 요구사항 (Supabase RLS 및 DB 설계)

### 4.1. 데이터베이스 스키마 설계안
* `profiles` (회원 정보): `id (UUID, auth.users.id 참조)`, `nickname`, `real_name`, `department`, `avatar_url`, `bio`, `role (user / admin)`, `created_at`
* `posts` (게시글): `id`, `user_id (profiles.id 참조)`, `content`, `image_urls`, `created_at`, `updated_at`
* `comments` (댓글): `id`, `post_id (posts.id 참조)`, `user_id (profiles.id 참조)`, `content`, `created_at`
* `likes` (좋아요): `post_id`, `user_id`, `created_at` (복합 기본키 설정)
* `tags` (태그): `id`, `name`
* `post_tags` (게시글-태그 맵핑): `post_id`, `tag_id`

### 4.2. 보안 및 RLS (Row Level Security) 정책
* 모든 테이블은 RLS를 활성화(`ALTER TABLE ... ENABLE ROW LEVEL SECURITY`)하여 비인가 사용자의 데이터 조작을 차단한다.
* **profiles**: 본인 데이터만 `UPDATE`, 전체 공개 `SELECT`.
* **posts**: 누구나 `SELECT`, 작성자 본인만 `UPDATE` 및 `DELETE`.
* **likes**: 누구나 `SELECT`, 본인 명의로만 `INSERT` 및 `DELETE`.
* **announcements (공지사항 테이블 또는 posts 내 type 구분)**: `SELECT`는 누구나, `INSERT/UPDATE/DELETE`는 `profiles.role = 'admin'`인 사용자만 허용.

## 5. TDD (테스트 주도 개발) 전략
* **원칙**: 모든 기능 개발 전 테스트 케이스를 우선 설계/작성한다.
* **테스트 분류**:
  1. **단위 테스트 (Unit Test)**:
     * 커스텀 가상 숫자 키패드의 입력 로직 테스트.
     * 해시태그 파싱 유틸리티 함수 테스트.
  2. **컴포넌트 테스트 (Component Test)**:
     * 로그인/회원가입 폼의 유효성 검사 및 에러 메시지 노출 테스트.
     * 무한 스크롤 트리거 및 데이터 추가 로드 동작 테스트.
  3. **통합 테스트 (Integration Test)**:
     * Supabase 클라이언트를 활용한 모의(Mocking) 인증 플로우 및 게시글 CRUD 흐름 테스트.
